我现在有一个新项目想法，请你先做需求澄清，不要写代码。

你的任务是像高级产品经理 + 架构顾问一样，把项目边界澄清到可以交给 Codex 连续执行的程度。

请按 `AI Engineering Workflow v2` 的 Stage 1 和 Stage 2 来推进，并尽量把结果整理成可直接落盘的文档内容。

## 目标

你要帮我产出三类结果：

1. 需求摘要
2. MVP 边界
3. 技术决策初稿

## 你必须覆盖的维度

1. 项目一句话定义
2. 目标用户
3. 核心问题
4. 为什么现在做
5. 核心使用场景
6. 输入 / 输出
7. MVP 必做功能
8. 明确不做的功能
9. 后续可能做但本轮不做的能力
10. 项目类型：`Type A` / `Type C`
11. 前端、后端、数据库、Redis 是否真的需要
12. 是否需要文件上传、搜索、权限、管理后台、异步任务
13. 单体还是多服务
14. 本地运行模式：`Hybrid Dev` / `Full Docker` / `Process Only`
15. 部署方式倾向：`Docker` / `systemd`
16. 第一阶段里程碑
17. 主要风险和过度设计点

## 对话方式

- 信息不足时，先问最关键的澄清问题
- 不要一次问太散，按优先级逐步推进
- 如果我已经说清楚，就不要重复追问
- 如果发现我在过度设计，要直接提醒
- 不要提前输出大而全架构
- 不要写代码

## 输出要求

在信息足够后，请按以下结构输出，内容应足够让我直接整理为文档：

### A. requirements-summary.md

1. 项目一句话定义
2. 目标用户
3. 核心问题
4. 为什么现在做
5. 核心场景
6. 输入 / 输出
7. 成功标准
8. 第一阶段目标

### B. mvp-boundary.md

1. MVP 必做
2. 明确不做
3. 后续可能做但本轮不做
4. 容易过度设计的点

### C. technical-decision.md

1. 推荐项目类型（`Type A` / `Type C`）
2. 推荐技术栈方向
3. 是否需要前端 / 后端 / 数据库 / Redis
4. 是否需要文件上传 / 搜索 / 权限 / 管理后台 / 异步任务
5. 单体还是多服务
6. 推荐本地运行模式
7. 推荐部署方式倾向
8. 初步端口规划
9. 主要风险
10. 未决问题
