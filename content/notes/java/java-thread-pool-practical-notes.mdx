## 为什么值得单独整理

线程池几乎出现在所有后端项目里，但真正出问题时，往往不是不会用，而是**没有在业务上下文里判断清楚参数和边界**。

## 我现在更关注的三个判断点

### 1. 任务是 CPU 密集还是 IO 密集

如果是 CPU 密集型任务，线程数通常不适合无限增加；如果是 IO 密集型任务，线程数则需要结合等待时间综合判断。

### 2. 队列长度是否可解释

无界队列看起来省心，但会把问题往后推。真正需要想清楚的是：

- 峰值流量大概有多高
- 最多允许积压多久
- 积压后谁来承担风险

### 3. 拒绝策略是否和业务一致

拒绝策略不是“防报错”的配置，而是系统在资源不足时的行为声明。是否丢弃、是否降级、是否回退到调用线程，应该和业务影响一起判断。

## 当前结论

在线程池配置上，我会先写清楚任务类型、容量预期和降级方式，再去填具体参数。这样得到的配置比“经验值套用”更稳定。
