# 资源下载方案

## 1. 文档目标

本文档用于定义 V0.1 下载资源的文件存放方式、元数据存放方式、命名规则以及“资源详情页 -> 下载文件”的映射方式。

## 2. 方案边界

V0.1 下载方案遵循以下约束：

- 不引入数据库
- 不引入对象存储
- 不引入下载鉴权
- 不通过接口动态签发下载地址
- 使用站内静态文件方案完成下载

## 3. 目录设计

### 3.1 元数据文件存放位置

资源元数据统一放在：

```text
content/downloads/
```

每个资源一个 `.mdx` 文件，负责维护：

- 资源名称
- 资源简介
- 资源类型
- 适用对象
- 包含内容
- 注意事项
- 下载路径

### 3.2 实体文件存放位置

资源实体文件统一放在：

```text
public/files/downloads/
```

推荐结构：

```text
public/
└─ files/
   └─ downloads/
      ├─ java-backend-interview-checklist/
      │  └─ java-backend-interview-checklist-v1.0.0.zip
      ├─ project-retro-template/
      │  └─ project-retro-template-v1.0.0.docx
      └─ backend-learning-roadmap-pack/
         └─ backend-learning-roadmap-pack-v1.0.0.zip
```

说明：下载文件放在 `public/files/downloads`，而不是直接放在 `public/downloads`，目的是避免与页面路由 `/downloads` 产生语义混淆。

## 4. 文件命名规范

### 4.1 资源元数据文件命名

- 文件名必须等于资源 `slug`
- 扩展名统一为 `.mdx`

示例：

- `content/downloads/java-backend-interview-checklist.mdx`
- `content/downloads/project-retro-template.mdx`

### 4.2 下载实体文件命名

推荐规则：

```text
<resource-slug>-<version>.<ext>
```

示例：

- `java-backend-interview-checklist-v1.0.0.zip`
- `project-retro-template-v1.0.0.docx`

约束：

1. 只允许小写英文、数字、中划线
2. 版本号建议显式写入文件名
3. 不使用中文文件名
4. 一个资源目录下，V0.1 推荐只保留一个当前有效下载文件

### 4.3 资源目录命名

- 目录名必须等于 `slug`
- 示例：`public/files/downloads/java-backend-interview-checklist/`

## 5. 资源详情页与下载文件的映射方式

### 5.1 映射原则

每个 DownloadResource 通过 frontmatter 中的 `downloadPath` 字段，显式映射到站内静态文件路径。

示例：

```yaml
downloadPath: /files/downloads/java-backend-interview-checklist/java-backend-interview-checklist-v1.0.0.zip
```

### 5.2 映射规则

1. `/downloads/[slug]` 页面根据 `slug` 找到对应的 `content/downloads/<slug>.mdx`
2. 读取 frontmatter 中的 `downloadPath`
3. 页面中的下载按钮使用该路径作为最终下载目标
4. 若 `downloadPath` 为空或文件不存在，该资源不应发布为 `published`

### 5.3 为什么使用显式映射

1. 资源详情与下载文件关系清晰
2. 即使文件名未来变化，也可通过 frontmatter 调整映射
3. 不依赖数据库查询或动态拼接规则
4. 便于本地校验和部署检查

## 6. 下载详情页展示规则

下载资源详情页应展示以下信息后，再提供下载入口：

- 资源名称
- 资源简介
- 资源类型
- 版本号
- 适用对象
- 包含内容
- 使用说明 / 注意事项
- 下载按钮

约束：

1. 列表页只展示摘要，不直接下载
2. 详情页必须先说明资源内容，再提供下载动作
3. 下载按钮文案默认可使用 `downloadLabel`，缺省时可回退到固定文案“立即下载”

## 7. 下载文件校验建议

阶段 3 开发和发布前，建议校验以下内容：

1. `slug` 与元数据文件名一致
2. 资源目录名与 `slug` 一致
3. `downloadPath` 路径可在 `public` 下找到对应文件
4. frontmatter 中的 `version` 与下载文件名中的版本号一致
5. `status=published` 的资源必须存在有效下载文件

## 8. 静态资源与下载资源的区分

建议区分两个目录用途：

- `public/images/*`：页面展示图片、封面图、图标
- `public/files/downloads/*`：可下载的实体文件

这样可以避免：

- 页面展示资源与下载文件混放
- 路径语义不清晰
- 后续部署时难以做目录级管理

## 9. V0.1 推荐方案结论

V0.1 推荐采用“MDX 元数据 + public 静态文件”的轻量下载方案：

1. 元数据放在 `content/downloads`
2. 实体文件放在 `public/files/downloads`
3. 通过 `downloadPath` 完成详情页与文件映射
4. 不引入接口层和权限控制

该方案最适合当前个人站点、低复杂度、可控文件量的阶段目标。
