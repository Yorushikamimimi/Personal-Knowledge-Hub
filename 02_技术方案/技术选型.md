# 技术选型

## 1. 文档目标

本文档用于明确 Personal Knowledge Hub V0.1 的技术选型、选型原因、边界约束与不采用方案，为阶段 3 的页面开发和内容接入提供统一依据。

## 2. 阶段 2 技术边界

V0.1 的技术方案只服务于“内容展示 + 资源下载 + 基础合规页面”这一范围，遵循以下边界：

- 使用 Next.js App Router 作为前台站点框架
- 使用 TypeScript 统一页面、内容模型与工具层类型定义
- 使用 Markdown / MDX 作为内容源
- 使用 `public` 目录承载静态资源与下载文件
- 不引入数据库
- 不引入后台 CMS
- 不实现登录、评论、搜索、投稿、接口层
- 不设计服务端数据写入链路

## 3. 最终选型结论

### 3.1 核心技术栈

- 前端框架：Next.js App Router
- 开发语言：TypeScript
- 内容驱动：Markdown / MDX
- 静态资源：`public`
- 部署运行：Node.js + PM2 + Nginx

### 3.2 内容组织方式

- `content/notes`：Note 内容源
- `content/downloads`：DownloadResource 元数据内容源
- `content/topics`：Topic 内容源
- `public`：图片、图标、下载文件等静态资源

## 4. 为什么 V0.1 采用 Next.js + TypeScript + MDX + public

### 4.1 为什么使用 Next.js App Router

1. V0.1 页面结构清晰，天然适合以文件路由管理固定页面范围
2. App Router 适合静态站点和内容型页面，可直接承接静态生成
3. 文章页和资源详情页可以基于 slug 在构建期生成，符合内容站稳定发布模式
4. 后续若要扩展 metadata、sitemap、布局、MDX 页面承载，也能在同一框架内完成

### 4.2 为什么使用 TypeScript

1. V0.1 已经明确了 `Note`、`DownloadResource`、`Topic` 三个核心内容模型，适合用类型约束统一字段
2. 页面字段、frontmatter 字段、工具函数输入输出都可以保持一致，降低后续字段漂移风险
3. 在没有数据库和 CMS 的情况下，类型系统本身就是最轻量的结构约束手段

### 4.3 为什么使用 Markdown / MDX

1. 项目本质是个人原创知识站，内容以笔记和资源说明为主，内容驱动优先级高于后台管理
2. Markdown / MDX 适合维护文章正文、下载说明、专题介绍等结构化文本
3. 内容可以直接随代码仓库版本管理，适合个人项目迭代
4. 不依赖数据库即可完成列表页和详情页展示
5. 后续若需要轻量增强，可在不破坏内容结构的前提下补充 MDX 组件能力

### 4.4 为什么使用 `public` 存放静态文件

1. V0.1 不引入对象存储和文件服务，`public` 是最直接且稳定的静态文件承载方式
2. 首页图片、站点图标、封面图预留、下载资源实体文件都可以统一管理
3. 构建后可随应用一并部署，减少额外运维复杂度
4. 对当前“低频更新、文件量可控”的阶段最合适

## 5. 为什么 V0.1 不采用数据库方案

### 5.1 业务复杂度不足以支撑数据库引入

V0.1 没有用户系统、没有写入型业务、没有后台管理，也没有复杂筛选与统计能力。页面内容完全可以通过 MDX 和静态文件描述，数据库不会带来实质收益。

### 5.2 数据维护方式更适合内容文件化

当前内容主体是文章、专题说明和下载资源说明，这些内容天然更适合文件维护，而不是表结构维护。

### 5.3 数据库会引入额外成本

如果在 V0.1 引入数据库，需要同时处理：

- Schema 设计
- 本地开发与线上环境一致性
- 迁移与备份
- 额外的连接配置与运维成本

这些成本与当前阶段目标不匹配。

### 5.4 文件化内容更适合个人站点

对于个人原创知识站，使用 Git 管理内容文件可以直接获得版本历史、回滚能力和审阅能力，整体更轻量、更稳定。

## 6. 为什么 V0.1 不采用后台 CMS

1. 当前没有多人协作编辑需求
2. 当前内容量和更新频率不需要专门内容后台
3. CMS 会引入额外账号、鉴权、部署与维护成本
4. 现阶段优先保证内容结构稳定，而不是编辑体验复杂化

## 7. 推荐技术原则

1. 优先静态化，尽量在构建期完成页面生成
2. 优先内容驱动，页面尽量从 frontmatter 和 MDX 正文读取数据
3. 优先类型约束，用 TypeScript 保持内容模型和页面字段一致
4. 优先轻量部署，不引入与当前规模不匹配的中间件
5. 优先可迁移，目录结构和字段规则要为后续扩展保留空间

## 8. V0.1 推荐依赖方向

V0.1 依赖应控制在“最小可用”范围，建议只围绕以下能力选取依赖：

- Next.js 基础运行
- React 基础运行
- TypeScript 类型支持
- MDX 解析与渲染
- Frontmatter 读取

说明：具体依赖包名和版本可在阶段 3 初始化工程时再确定，但不应引入数据库、认证、搜索、CMS、上传等相关依赖。

## 9. 承接说明

阶段 3 页面开发应直接建立在本选型之上：

1. 路由按 App Router 组织
2. 页面数据按 MDX 内容读取
3. 下载文件按 `public` 静态目录映射
4. SEO 与 metadata 按页面类型统一生成
